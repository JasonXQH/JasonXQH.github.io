manage.py 用来执行一些管理任务，比如开启服务器，发布数据等等

外部的vidly师我们创建的，用来装这个项目的容器。里面的vidly是一个包，是整个项目的核心

里面有一个__init__ 告诉python这是一个包。settings.py中是一些项目相关的配置，url定义项目所有的url，wsgi.py是网络服务网关接口的缩写，这个文件的功能是创建一个标准的接口，链接python应用和网络服务器。

python manage.py runserver 启动服务

步骤详解

创建应用

一个Django工程之中有很多的app，但这些app不代表整个程序，他们只专注于某个功能。比如我们想要创建一个类似于Amazon的程序。那么我们就要这样创建

以这样的设计，我们可以将大型的应用，分解为更好管理的模块。

我们新建一个movies模块python manage.py startapp movies

在这个文件夹里，有

__init__.py 是把标志这个为一个模块

admin.py 定义了管理电影的页面长什么模样

apps.py 存放App的信息的，类似config

models.py 定义了app中可能要用到的类，如movieclass,genreclass

tests.py 存放自动测试的程序

views.py 存放了视图函数，也就是接受请求，返回反馈

Models

接下来我们要创建movies应用的模型了。这个model代表应用数据的类。在movie这个app中可以有genre类和movie类，现在我们创建他们

在movies文件夹下的models.py中我们来新建models，首先让他继承自models中的Model类。在Django中有个类叫做哦django.db 。这个类封装了所有处理数据库的函数。这个类中有个模块叫做models，models模块中有个类叫做Model，这个类中封装了很多向数据库存储Model对象的方法，已经处理、筛选它们

让Genre类继承自Model类，可以不用再为了将Genre类保存到数据库写代码了。Django会自动完成。

models这个模块中有很多字段类型， CharField、IntegerField、FloatField、BooleanField 等

在Genre类中我们设置了name字段，在Movie类中我们设置了title，release_year, number_in_stock ,daily_rate , genre类型。其中genre类型需要和Genre类建立联系，这里用了model.ForeignKey 方法，第一个参数是Genre类，第二个参数是on_delete 也就是删除了一个类之后进行的操作，在这里，我们设置为models.CASCADE,意思是当删除一个类之后，这个类中的所有的电影信息都将被删除

Migrations

我们创建了model类，现在我们可以将model存储到数据库当中

找到db.sqlite3 这个文件，是一个空数据库，这个是Django自动为我们创建的。sqlite是一个轻量级数据库，很适合我们做小型项目的时候使用，但是他不太适合企业级应用，它不支持安全和并发。

现在我们先存储到sqlite，然后用mysql来代替

我们先来下载 DB Browser for sqlite 这个软件，这里可以创建新的库或者打开已经有的库

打开db.sqlite3之后，我们发现这是一个空的数据库，现在我们想把电影和分类保存在库中。

当使用Django的时候，我们不用去设计数据表。我们可以用Django给予我们写好的model自动创建。每当我新建或者修改一个model的时候，我们会告诉Django，用现在的模型和数据库比较。Django会检查数据库，找出我们已经有的表，计算模型类和数据表之间的区别。基于计算结果，他会创建一个迁移文件。签移文件实际上是一个python文件，运行它的时候会更新sqlite中的数据表

首先，我们要把movies应用注册到Django，打开配置文件夹vidly,里面有个settings.py 我们看到 INSTALLED_APPS 这里显示的就是已经注册了的app

auth是验证app

contenttypes用来创建基于类之间的通用关系

sessions：临时保存客户信息的框架

messages: 比如用户创建了一个新电影，我们就重定向它，并显示一个message

staticfiles： 用来存储css等静态文件

最后我们把movies这个app注册进去，这个app放在movies文件夹下的apps.py文件内，叫做MoviesConfig

然后我们终端输入 python manage.py makemigrations

我们看到这个migrations文件夹下的0001_initial.py文件

然后我们终端输入 python manage.py migrate 将所有的migration文件全部运行，运行完成后再次打开数据库我们的sqlite中就有表格和索引了。 里面有个django_migrations数据表，记录了我们操作的migration文件和时间

Changing the Models

我们现在向Movie这个model中添加一个创建时间字段。类型DateField，值是创建这条信息的时间

这里我们不想用python的内置标准库也就是datetime.now() 但是需要注意，这个时间日期对象是不关心时区的。更好的方式是使用Django中的时区对象，当以后需要将时间显示给用户的时候，会自动切换成用户所在地区的时间。

还需要注意，这里传入的是一个引用而不是调用这个方法。因为当我们写的是 timezone.now()的时候，其实我们把默认值设置成了一个常量，也就是生成migration的时间。而我们需要动态生成。

python manage.py migration 生成文档之后，我们看看operations

注意，我们千万不能删除任何一个migration文件，因为每个migration文件都是基于上一个文件基础上的修改！当我们删掉了migration文件的时候，就会缺失了中间某一步

python manage.py migrate之后，这条字段就被添加到数据库当中了

如果我们想看每一次migrate操作时，Django向数据库发送的sql语言。那么我们可以用这样的语句

python manage.py sqlmigrate movies 0001(migration文件的标号)

接下来我们来创建一个api，这样，当其他的网站向我们发送一个请求，我们会以json的格式返回电影信息。

这样他人可以获得这些内容，服务于他们的应用。

要在Django中创建api，有两个比较流行的框架： djanngo-tastypie 和 djangorestframework（DRF）这里我使用前者

我们新建了一个api，这个应用的功能就是提供api

在api文件夹下的models.py 我们新建一个MovieResource的model，用它来代表RESTful api中的电影资源的概念。我们需要让他来继承自DJango-tastipie 中的 ResourceModel类

tastypie会搜索这个类当中的子类meta，meta类当中定义了电影资源的一些源数据。其中国需要设置一些tastiepie关注的字段：

第一个字段是queryset,在这里我们把它设置为获得所有电影列表的查询。Movie.objects.all() 返回的也是一个查询，他不会去数据库取回所有的电影。这个查询是在未来的某时刻才执行的，所以也叫作懒加载（lazy loading)

第二个字段是resource_name 这决定了我们API实际的地址 也就是 http://127.0.0.1:8000/api/movies/ 中的movies
